# Guía para release

Esta guía es para quien publica una versión.

## Tu objetivo

- Publicar solo cuando el cambio esté validado.
- Reducir riesgo de entrega.
- Dejar clara la versión y el impacto del release.

## Flujo de trabajo

1. Revisa el `release approval request`.
2. Confirma que la validación esté completa.
3. Verifica el alcance y los riesgos.
4. Revisa `CHANGELOG.md`.
5. Corta el tag de versión correspondiente.
6. Publica el release si aplica.

## Qué debes cuidar

- El release debe estar ligado a una versión concreta.
- Si hay riesgo alto, pide aprobación explícita.
- Si algo quedó pendiente, no publiques todavía.

## Qué no debes hacer

- No cortar releases desde una rama no revisada.
- No publicar sin saber qué cambió.
- No dejar el changelog vacío cuando hay impacto visible.
